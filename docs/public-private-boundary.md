# Public / Private Boundary

The public repo should teach a method. It should not publish the private
operating history that inspired the method.

## Public

Safe public material:

- template files
- schemas
- fictional examples
- generic maintainer workflows
- source-check rules
- write-back rules
- contribution and release checklists

## Private

Do not publish:

- private chats
- private task boards
- raw logs
- credentials, tokens, cookies, or API keys
- customer, patient, financial, or family data
- real screenshots that show accounts, messages, names, or orders
- internal project names that identify private work

## Practical Rule

If the material loses its value when names, dates, accounts, and private context
are removed, it belongs in a private workspace, not in this public kit.

If the material still helps someone run a better agent workflow after all
private details are removed, it can become a public template or synthetic
example.
